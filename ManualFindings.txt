── BUGS ─────────────────────────────────────────────────────────────────────────

BUG-1: Approval-requiring tools block the SSE stream indefinitely
  Tests 6,7,8,14 all timed out at ~120s. When the agent calls az_cli or run_shell,
  the SSE stream hangs at `approval_required` and the client can't approve
  because it's still reading the same stream. The approval event arrives but
  events=[]: the test client never saw it because httpx iter_lines() didn't
  flush the approval event before blocking on the next read.
  ROOT CAUSE: The test script's parse_sse approach reads the full stream
  synchronously. But more importantly, the backend blocks the SSE generator
  on `await wait_for_approval()`, so the approval_required event and
  subsequent events are in the same HTTP response. The client must approve
  via a *separate* HTTP call while the stream is still open. The test script
  (and potentially the terminal client) need to handle SSE in a separate thread.
  IMPACT: Terminal client will also hang on approval if not threaded.
  FIX NEEDED: Terminal client main.py must read SSE in a background thread
  so it can prompt for approval and POST the resolution while the stream
  continues. The e2e test script also needs this pattern.

BUG-2: Content filter on repeated characters (5000x 'x')
  Test 20 sent 5000 'x' chars — Azure OpenAI flagged it as violence:high and
  returned content_filter error. The backend propagates this as an SSE error
  event but doesn't sanitize or provide a user-friendly message.
  FIX NEEDED: Catch content_filter errors in orchestrator and return a
  friendlier message like "Your message was flagged by content filtering.
  Please rephrase." Also consider adding input length validation at the API
  layer (e.g., max 4000 chars).

── ISSUES ───────────────────────────────────────────────────────────────────────

ISSUE-1: Agent never calls az_cli proactively for simple queries
  Tests 6 ("What is my current Azure CLI account?") and 7 ("What is the date?")
  should have triggered az_cli or run_shell calls, but they timed out — meaning
  the agent *did* try to call them (approval-blocked). This is actually correct
  behavior — it's using the right tools. The problem is only the test harness.
  STATUS: Working as designed, just needs the approval flow fix (BUG-1).

ISSUE-2: Cost query (Test 8) hangs on approval
  "Show me the cost summary for last month" — the agent tried to call a tool
  that requires approval (likely az_cli for cost management). Same as BUG-1.
  The agent correctly follows the tool hierarchy (tried az_resource_graph
  first per the hierarchy, then fell back to az_cli for cost data which
  requires approval). This is correct behavior blocked by BUG-1.

ISSUE-3: Agent calls read_learnings + fetch_ms_docs before every Azure query
  Tests 4,5,9,10,13,16 all show read_learnings as the first tool call, then
  often fetch_ms_docs before the actual query. This adds 2-4s overhead per
  request. Consider:
  a) Injecting learnings into the system prompt (already done — so the
     explicit read_learnings call is redundant)
  b) Only calling fetch_ms_docs when the agent doesn't know the syntax
  IMPACT: Performance — avg 10-15s for Azure queries vs 4-6s for KB queries.

ISSUE-4: Test 15 ("What skills are available?") doesn't know about skills
  The agent searched the KB 3 times but couldn't find skill info. Skills are
  a system concept, not a KB topic. The agent should use introspection to
  list available tools/skills, or we need a tool that lists them.
  FIX: Add a `list_skills` tool or add skill info to the system prompt.
